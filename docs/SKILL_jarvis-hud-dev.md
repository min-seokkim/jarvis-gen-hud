# jarvis-hud 개발 Workflow 스킬

이 프로젝트의 기능을 만들 때 항상 이 절차를 따른다. 목표는 속도가 아니라 **평가 핵심(generative HUD)을 보호하면서 매 단계 검증된 결과를 내는 것.**

## 0. 시작 전 맥락 고정
다음을 읽고 시작한다(없으면 사용자에게 위치 요청):
- `docs/기획서.md` — 주제·핵심기능·기술스택·화면흐름
- `docs/agent-workflow.md` — 작업 쪼개기 기준·프롬프트 패턴·체크포인트
- `jarvis_handoff.md` — 설계 배경·관통 원칙
- 진행 중이면 라이브 로드맵(M0–M6)

## 1. 불변 원칙 (어길 것 같으면 멈추고 사용자에게 확인)
- **0부터 짓는 건 generative HUD 하나.** 두뇌·음성·도구 합성은 기존 완성형에 위임.
- **계산은 deterministic, LLM은 오케스트레이션.** 숫자/계산을 LLM이 지어내게 두지 않는다.
- **자유 생성 + 디자인 언어 제약.** 생성 UI는 디자인 토큰·허용 컴포넌트 스코프만 사용.
- **음성 즉답은 클라우드 임계경로에 두지 않는다.** 로컬이 즉시 답, HUD는 병렬 스트리밍.

## 2. 절차
1. **쪼개기** — 요청을 기능 단위(데모 가능한 수직 슬라이스) 또는 컴포넌트 단위로 분해. 한 번에 하나만.
2. **낯설면 de-risk** — 처음 쓰는 기능/라이브러리는 throwaway 스파이크로 "되는가"만 먼저 증명.
3. **생성** — 출력 형태를 명시해 요청(단일 파일/패치/표). TS, 외부 라이브러리 임의 추가 금지.
4. **검증(아래 체크포인트)** — 통과 못 하면 다음으로 안 넘어간다.
5. **통합** — feature 브랜치 커밋, `feature/* → dev` PR. 커밋에 `#이슈번호`.

## 3. 검증 체크포인트 (이 결과를 그냥 믿지 말 것)
- 빌드/타입 통과(tsc 0), 실제 렌더/실행 확인, 에러 경계 동작.
- **HUD:** 생성 JSX가 디자인 토큰·허용 컴포넌트만 쓰는가 / iframe 격리 안전한가 / 자기치유 재시도 cap·폴백 있는가 / 본 적 없는 작업에 새 HUD가 실제로 나는가.
- **정확성:** 계산·시뮬은 deterministic 코드가 했는가, 수치 재검증했는가.
- **성능:** 음성 왕복 실측(목표 0.5s), 음성+HUD 동시성.
- **보안:** API 키 프론트 미노출(Caddy 주입), TLS·basic-auth, 0.0.0.0 직노출 없음.

## 4. 출력 규약
- React/HUD는 단일 `.tsx`로, props 타입 명시, 사용 예시 포함.
- 수정은 가능한 한 패치(diff)로 제시.
- 불확실하면 "모른다"고 말하고 출처/근거를 단다. 추측으로 숫자·API를 지어내지 않는다.
