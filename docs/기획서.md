# 프로젝트 기획서 — 자비스형 엔지니어 도우미 (Generative HUD)

---

## 1. 서비스 주제 및 핵심 기능

### 한 줄 정의
영화 아이언맨의 자비스를 모델로 한, **음성으로 대화하고 작업 맥락에 맞는 UI를 그 자리에서 생성해내는 엔지니어 통합 작업 도우미.**

### 무엇을 만드는가
자비스는 일정·이메일을 챙기는 비서가 아니라 토니 스타크의 **R&D 워크벤치 전체를 운영하는 엔지니어링 파트너**였다. 이 프로젝트는 그 본질 — "엔지니어와 함께 일하는 도우미" — 을 지향한다.

### 핵심 기능: Generative HUD
자비스의 핵심은 멋진 고정 인터페이스가 아니라 **작업이 바뀌면 인터페이스가 따라 생성된다**는 점이다. 엔지니어링은 작업마다 필요한 표현이 다르므로(회로면 회로도, 신호면 파형, 데이터면 차트) 고정 UI 하나로 덮을 수 없다. 그래서 **"맥락을 읽고 거기 맞는 UI를 LLM이 실시간 생성"하는 generative HUD**가 이 프로젝트의 독창적 핵심이며, **0부터 직접 짓는 유일한 부분**이다.

| # | 핵심 기능 | 설명 |
|---|---|---|
| 1 | **제약 JSX 실시간 생성** | 작업 맥락 → LLM이 디자인 시스템 컴포넌트·토큰만으로 JSX 생성 |
| 2 | **샌드박스 렌더 + 자기치유** | react-live/Sandpack에서 iframe 격리 렌더, 에러 시 에러를 다시 먹여 재생성 |
| 3 | **본 적 없는 작업의 HUD 발명** | 미리 만들지 않은 작업을 던져도 인터페이스가 새로 태어난다 |
| 4 | **음성 + HUD 동시성** | 음성으로 답하는 *동안* 화면에 HUD가 생성된다 |

### 프로젝트 성격
백엔드/AI 모델링 프로젝트가 아니다. **평가 대상은 generative HUD를 구현한 React 프론트엔드.** 두뇌·음성·도구 합성처럼 잘 만들어진 것은 적극적으로 갖다 쓰고, 역량은 프론트엔드에 집중한다.

## 2. 기술 스택 및 선택 이유

핵심 원칙: **갖다 쓸 수 있는 건 갖다 쓰고, 0부터 짓는 것은 generative HUD 하나.** 백엔드 늪에 빠지면 정작 평가 대상인 프론트엔드를 못 끝낸다.

| 영역 | 선택 | 선택 이유 |
|---|---|---|
| **프론트엔드** | React (Vite + TS) 반응형 웹앱 | 폰/PC를 한 앱으로 커버. generative HUD가 핵심이라 컴포넌트·상태 관리가 강한 React |
| **HUD 렌더링** | 제약 JSX 샌드박스 (react-live/Sandpack) | LLM이 만든 임의 JSX를 **안전하게** 렌더해야 함 → iframe 격리 + 디자인 시스템 컴포넌트만 스코프 노출 + 자기치유 루프 |
| **두뇌/에이전트** | Hermes Agent (Nous Research) | 자율 스킬 생성·영속 메모리·도구 합성을 **이미** 내장 → 직접 안 짜도 됨. OpenAI 호환 API 서버 내장이라 React가 표준 SSE로 바로 연결 |
| **두뇌 모델 (역할 하이브리드)** | 빠른 메인(Claude Haiku 4.5 / GPT-5.4-mini) + 강한 delegation(Claude Opus 4.6 / GPT-5.5) | **단일 Hermes 브레인.** 메인=대화·음성·즉답·dispatch, delegation=추론·JSX 생성·코딩. 프로바이더 **OpenAI·Anthropic만**, 호출부 모델 무관 추상화(교체 가능) |
| **STT** | faster-whisper (로컬) | 한국어+영어 멀티링궐, 네트워크 0 |
| **TTS** | ElevenLabs (Flash v2.5) | 자비스 보이스, multilingual 한국어, 저지연 |
| **라우팅** | 역할 하이브리드 (단일 브레인) | 빠른 메인이 음성·즉답·dispatch, 추론은 강한 delegation에 위임. **음성 즉답을 클라우드 임계경로에 두지 않음**(병렬). |
| **디자인 언어** | 자비스 클리셰 | 청록 홀로그램·검은 배경·파랑(안정)↔빨강(경고) 상태 매핑 → 자유 생성 + 디자인 토큰 강제로 일관성 유지 |

> **용어 메모 — "OpenAI 호환 API":** OpenAI사를 호출한다는 뜻이 아니라, **Hermes가 띄운 로컬 API 서버**(`/v1/chat/completions`)가 OpenAI와 **같은 규격**이라는 의미다. 덕분에 React가 표준 OpenAI 형식으로 바로 붙고, 나중에 base URL만 바꿔 다른 OpenAI 호환 서버로 교체할 수 있다. (두뇌=클라우드 모델이 현재 OpenAI사인 것과는 **별개** — 두뇌를 바꿔도 이 API 서버는 계속 OpenAI 호환.)

> **갱신 메모 — 라우팅(ADR 0001 우선):** 초기 '음성=로컬 Qwen / 화면=클라우드' 출력양식 라우팅은 두-브레인 코히어런스 문제로 **폐기**됨. 확정 = **단일 Hermes 브레인 + 역할 하이브리드**(빠른 메인=음성·즉답·dispatch, 강한 delegation=추론). 로컬 LLM(Qwen/Ollama) 안 씀 — STT(faster-whisper)·TTS(ElevenLabs)는 유지. 근거: `docs/decisions/0001-framework-hermes.md`.

### 관통하는 설계 원칙
- **계산은 deterministic, LLM은 오케스트레이션.** 엔지니어링은 환각이 곧 사고 → 실제 계산은 코드/도구가, LLM은 "무엇을 실행하고 어떤 HUD로 보여줄지"만 지휘.
- **정적 카탈로그가 아니라 합성 엔진.** UI도 도구 어댑터도 미리 다 만들지 않고 필요할 때 생성.
- **자유 생성 + 디자인 언어 제약.** "아무렇게나 생성" 방지. 자비스스러운 디자인 언어 유지.

### 배포 구조 (3주 공통 요구: 배포 가능)
집 RTX 4080 워크스테이션이 서버(Hermes + 음성 파이프라인 + React 호스팅), 데모장에선 노트북 브라우저로 접속. 연결은 포트포워딩 + DDNS + Caddy(TLS·basic-auth, API 키는 서버측 주입). 배포 파일: [`deploy/`](../deploy).

## 3. 화면 흐름 / 페이지 구성 초안

### 페이지 구성 (단일 반응형 웹앱)
화면은 크게 둘로 나뉜다.

```
┌─────────────────────────────────────────────┐
│  상태바: 자비스 상태(대기/청취/사고/렌더/경고) │
├──────────────────┬──────────────────────────┤
│                  │                          │
│   대화 패널       │     HUD 캔버스           │
│  (음성/텍스트     │  (LLM이 생성한 JSX가     │
│   대화 스트림)    │   여기 실시간 렌더)       │
│                  │                          │
├──────────────────┴──────────────────────────┤
│  입력: 마이크 버튼(음성) / 텍스트 입력        │
└─────────────────────────────────────────────┘
```
- **모바일:** 대화/HUD를 탭 또는 세로 스택으로 전환.
- **PC:** 좌우 분할(대화 | HUD 캔버스).

### 핵심 인터랙션 흐름
```
사용자 발화/입력
   │
   ├─ faster-whisper STT (로컬) → 오케스트레이터(워크스테이션 Python)
   │
   └─ Hermes /v1  (단일 브레인)
        ├─ 빠른 메인: 짧은 음성 즉답/ack → ElevenLabs TTS 즉시 재생  ← 임계경로에 안 묶임
        └─ (화면 필요 시 병렬) 강한 delegation: 제약 JSX 생성
                 └─▶ 샌드박스 렌더 (에러 시 자기치유) → HUD 캔버스에 표시
```
음성과 HUD가 **동시에** 일어나는 게 핵심 — 음성만 답하면 일반 음성비서로 보인다.

### 화면 상태 (자비스 디자인 언어)
| 상태 | 표현 |
|---|---|
| 대기(idle) | 청록 은은한 글로우 |
| 청취(listening) | 파형/펄스 애니메이션 |
| 사고·생성(thinking/rendering) | HUD 캔버스에 스켈레톤 → 점진 렌더 |
| 정상/경고 | 파랑(안정) ↔ 빨강(경고/실패) 상태 매핑 |
| 자기치유(self-heal) | 렌더 에러 시 조용히 재생성, n회 실패하면 폴백 카드 |

### 데모 시나리오 (페이지가 증명할 것)
1. **보편 후크:** "프로젝트 빌드 상태 보여줘" → 음성 짧게 답 + 화면에 빌드 진행 HUD(진행 막대·단계·빨강 실패 단계) 생성. 도메인 몰라도 즉시 이해.
2. **마법(핵심 증명):** 미리 만들지 않은 작업을 던졌을 때 인터페이스가 새로 태어나는 순간.
