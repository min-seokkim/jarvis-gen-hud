# Agent 개발 Workflow 초안

---

## 0. 전체 루프 (한 사이클)

```
① 맥락 고정   → 에이전트에게 관련 문서/파일을 컨텍스트로 준다 (AGENTS.md, handoff, 기획서, 로드맵)
② de-risk     → 낯선 부분은 throwaway 스파이크로 먼저 검증 (커밋 전에 미지수를 죽인다)
③ 쪼개기      → 작업을 기능/컴포넌트 단위로 분해해 1개씩 요청
④ 생성        → 프롬프트 패턴으로 요청 → 코드/문서 생성
⑤ 검증        → 체크포인트로 내가 직접 판단 (빌드/렌더/계산/디자인/보안)
⑥ 통합·커밋   → feature 브랜치에 커밋, feature→dev PR
⑦ 회고·기록   → 막힌 지점·교훈을 메모리/회고에 남겨 다음 사이클 개선
```
핵심 신념: **"갖다 쓸 수 있는 건 갖다 쓰고, 계산은 deterministic, LLM은 오케스트레이션."** 에이전트는 속도를 주지만 **판단(⑤)은 내가** 한다.

## 0.5. 도구 역할 분담 (Cowork ↔ Claude Code)

빌드 단계의 두 에이전트를 "각자 잘하고 접근 가능한 것"으로 가른다.

| 영역 | 담당 |
|---|---|
| 구현 — 컴포넌트·HUD 샌드박스·API 배선·빌드/테스트/버그수정, **git·브랜치·PR·이슈** | **Claude Code** (repo 안) |
| 기획·로드맵·리서치·의사결정, 문서(기획서·회고·위키), **디자인 토큰/HUD 스펙**, 데모(슬라이드)·고수준 리뷰 | **Cowork** |
| 최종 실측·판단·승인 | **사람(나)** |

핸드오프: **Cowork** 가 스펙·이슈·디자인 토큰·결정 근거 → **Claude Code** 가 repo에서 구현·커밋·PR → **Cowork** 가 리뷰·다음 리서치·로드맵/회고 갱신.

> **Claude Code가 일차로 읽는 기준 문서는 [`AGENTS.md`](../AGENTS.md)** 이며, `CLAUDE.md`가 이를 참조(`@AGENTS.md`)한다. 즉 이 workflow의 규칙이 그 문서를 통해 Claude Code의 작업 기준이 된다.

## 1. 작업 단위 쪼개기 기준

### 기능 단위 (수직 슬라이스) — "한 번에 도는 최소 루프"
- 사용자에게 의미 있는 동작 하나가 끝에서 끝까지 도는 단위로 자른다.
- 예) "텍스트 명령 → Hermes 스트리밍 응답이 화면에 뜬다", "빌드 상태 HUD 1종이 생성된다".
- 기준: **데모 가능한가? Exit 조건이 한 문장으로 떨어지는가?** → 그러면 한 기능 단위(= 한 Issue).
- 로드맵 M0~M6의 각 마일스톤이 대략 이 단위.

### 컴포넌트 단위 (수평 슬라이스) — 기능 안에서 더 쪼갤 때
- 재사용 컴포넌트/모듈로 분해. 예) HUD 프리미티브(ProgressBar, Gauge, StatusPanel, Chart, Alert), 샌드박스 래퍼, 디자인 토큰, 음성 파이프라인의 STT/LLM/TTS 어댑터.
- 기준: **다른 기능에서도 재사용되는가 / 독립적으로 테스트되는가** → 그러면 별도 컴포넌트(= sub-task).

### 쪼개기 규칙 (운영)
- 한 PR = 한 기능 단위 (리뷰 가능한 크기). 너무 크면 컴포넌트 단위로 분할.
- 에이전트에 한 번에 한 단위만 요청한다 (한 프롬프트에 기능 여러 개 X).
- 평가 핵심(M3·M4 generative HUD)은 더 잘게 쪼개 매 단계 검증.

## 2. AI 요청 프롬프트 패턴 초안

공통 골격: **맥락 → 목표 → 제약 → 출력 형태 → 검증 기준.**

### 패턴 A — 맥락 주입형 (작업 시작)
```
[맥락] 첨부한 기획서/로드맵을 참고해. 우리는 generative HUD를 만든다.
[목표] M2의 HUD 프리미티브 중 ProgressBar 컴포넌트를 만들어줘.
[제약] 디자인 토큰(청록/검정/파랑↔빨강)만 사용, 외부 라이브러리 금지, TS.
[출력] 단일 .tsx 파일 + 사용 예시. props 타입 명시.
[검증] 내가 Storybook에서 상태별(정상/경고)로 확인할 수 있게.
```

### 패턴 B — de-risk 스파이크형 (낯선 기능)
```
이건 버릴 코드다. 최소한으로 "X가 되는가"만 증명해줘.
성공 기준은 한 문장: "____ 가 화면/로그에 뜬다". 예쁘게 만들지 말 것.
```

### 패턴 C — 생성→검증 루프형 (HUD JSX 생성처럼 결과가 불확실할 때)
```
[생성] 이 작업 맥락에 맞는 HUD JSX를 디자인 시스템 스코프만으로 생성해.
[자기검증] 렌더 에러가 나면 에러 메시지를 다시 받아 스스로 수정해(최대 n회).
[보고] 최종 JSX와 함께, 어떤 토큰/컴포넌트를 썼는지 한 줄 요약.
```

### 패턴 D — 의사결정/조사형 (스택·도구 선택)
```
[선택지] A vs B vs C 중 우리 제약(2주, 프론트 평가, 저지연)에 뭐가 최선?
[요청] 각각 장단점 + 추천 1개 + 추천이 틀어질 조건(언제 다른 걸 골라야 하나).
근거를 출처와 함께. 모르면 모른다고.
```

### 패턴 E — 검증/리뷰 요청형
```
방금 만든 코드/계산을 의심하는 입장에서 검토해줘.
특히 [환각 가능 지점 / 엣지케이스 / 보안]을 짚고, 테스트로 증명해.
```

### 프롬프트 운영 원칙
- **항상 맥락 문서를 먼저 준다** (기획서·로드맵·runbook). 맥락 없는 요청 금지.
- **출력 형태를 명시** (파일 1개? 패치? 표? 명령어?).
- **"모르면 모른다고, 출처 달고"** 를 요구해 환각을 줄인다.
- 큰 요청은 패턴 B(스파이크)로 먼저 찔러보고 패턴 A로 본 구현.

## 3. 내가 직접 검증/판단해야 할 체크포인트

에이전트가 만든 결과를 **그대로 믿지 않는다.** 아래는 내가 눈/손으로 확인할 목록.

### 빌드·동작
- [ ] 빌드/타입 통과하는가 (`npm run build`, tsc 에러 0)
- [ ] 실제로 렌더/실행되는가 (스크린샷 또는 직접 확인)
- [ ] 에러 경계가 동작하는가 (잘못된 입력에 앱이 죽지 않는가)

### Generative HUD (평가 핵심)
- [ ] 생성된 JSX가 **디자인 토큰·허용 컴포넌트만** 쓰는가 (막 생성 아닌가)
- [ ] 샌드박스가 **iframe 격리**되어 안전한가 (호스트 앱에 영향 없는가)
- [ ] 자기치유 루프가 무한 재시도/폭주하지 않는가 (재시도 cap·폴백 있는가)
- [ ] "본 적 없는 작업"에 정말 새 HUD가 나는가 (템플릿 우회가 아닌가)

### 정확성 (엔지니어링 = 환각이 사고)
- [ ] 계산·시뮬 결과는 **deterministic 코드/도구**가 했는가 (LLM이 숫자 지어내지 않았나)
- [ ] 수치는 직접/프로그램으로 재검증했는가

### 음성·성능
- [ ] 음성 왕복 실측 지연이 목표(0.5s)에 얼마나 가까운가 (localhost + 원격 둘 다)
- [ ] 음성과 HUD가 **동시에** 일어나는가 (음성이 클라우드 임계경로에 묶이지 않았나)

### 보안·배포
- [ ] 공개 종단에 **API 키가 프론트에 노출되지 않는가** (Caddy 서버측 주입 확인)
- [ ] TLS·basic-auth가 걸려 있는가, 0.0.0.0 직노출은 없는가
- [ ] 배포가 한 번에 재현되는가 (compose up 한 방)

### 데모 관점
- [ ] 청중이 도메인 몰라도 이해되는가
- [ ] "마법" 순간이 데모에 실제로 보이는가
- [ ] 네트워크 죽어도 폴백(백업 녹화)이 있는가

## 4. 3주에 걸쳐 발전시킬 것
- 효과 좋은 프롬프트 패턴·규칙은 [`AGENTS.md`](../AGENTS.md)에 반영해 Claude Code의 작업 기준으로 굳힌다.
- 반복되는 검증을 자동화(테스트/스크린샷/린트)해 체크포인트를 줄인다.
- 막힌 지점·교훈을 회고에 남겨 다음 사이클 프롬프트를 개선한다.
