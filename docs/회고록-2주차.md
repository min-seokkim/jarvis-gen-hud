# 2주차 회고록 (KPT)

## 이번 주 한 일
- **M1 — 앱 셸 + 스트리밍 대화:** Vite+React+TS 스캐폴딩(ESLint/Prettier/dev proxy), Hermes SSE 스트리밍 클라이언트(fetch+ReadableStream), 반응형 셸 + 토큰 스트리밍 + 에러 UI. 프록시로 Hermes auth 주입(Vite 8 `headers`), `/v1` 프록시에서 Origin 헤더 스트립(프론트·Caddy 양쪽), 프론트의 시스템 프롬프트 주입 제거.
- **두뇌 정비:** JARVIS 페르소나(SOUL.md), ADR 0001~0003(프레임워크=Hermes, 타당성 감사, config 변경)로 결정 근거 기록. orchestrator 하이브리드 voice-routing stub.
- **M2 — 디자인 시스템:** HUD 디자인 프리미티브 + 프리미티브 갤러리, design-system 스펙.
- **M3 — generative HUD 코어(평가 핵심):** react-live 런타임 + Playwright e2e 도입, deterministic build-status 데이터 소스, **제약 JSX 생성 + 자기치유 계약**, react-live 샌드박스 렌더, 생성·자기치유를 대화 흐름에 배선, 끝-끝 e2e + Vitest 컴포넌트 테스트 셋업.
- 결과: "텍스트 명령 → 스트리밍 응답 + 맥락에 맞는 HUD가 생성·렌더"되는 **핵심 수직 슬라이스가 한 바퀴** 돌게 됨.

## Keep (잘한 것, 유지)
- **수직 슬라이스로 한 바퀴를 먼저 완주했다.** M1(대화)→M3(HUD 생성·렌더)까지 데모 가능한 최소 루프를 우선 확보 — 평가 핵심을 가장 먼저 살아 있게 만든 게 옳았다.
- **LLM 출력을 "계약 + 자기치유 + 테스트"로 감쌌다.** 제약 JSX 계약과 자기치유 루프를 코드로 박고, Playwright e2e + Vitest로 핵심 루프를 회귀로 잠갔다. 생성물의 불확실성을 구조로 방어.
- **보안 원칙을 M1부터 지켰다.** 키를 프론트에 안 넣고 프록시/Caddy 서버측 주입. 막판이 아니라 처음부터.
- de-risk를 먼저: react-live에 하드코딩 JSX부터 찔러보고 본 구현으로.

## Problem (문제, 아팠던 것)
- **"배선" 버그에 시간이 샜다.** Vite 8 프록시 auth, Origin 헤더 스트립(프론트·Caddy 두 곳), 응답 공백/개행 스트립 등 — 1주차 Try였던 "인프라 최소경로 통과"를 또 완전히 못 지켰다.
- **책임 경계를 늦게 정했다.** 프론트가 시스템 프롬프트를 주입하다 페르소나/두뇌 책임이 흐려져 되돌렸다(refactor). "프론트=표현, 두뇌=내용"을 먼저 못 박고 시작.
- M1(06/04)과 M2·M3(06/09) 사이 착수 공백 — 코어 진입이 며칠 늦었다.

## Try (다음 주에 시도할 것)
- **HUD 생성의 안전·품질을 최우선으로.** 지금 react-live는 같은 문서 인라인 실행 → **샌드박스 격리(iframe)와 validator 강화**가 다음 주 1순위.
- 프롬프트 "계약"을 더 좁히고 엣지 케이스를 테스트로 고정.
- 책임 경계(표현 vs 내용)를 AGENTS.md에 먼저 박고 작업 시작.

## 학습 목표 대비 (2주차: 핵심 기능 FE-BE 동작 + 수직 슬라이싱)
- ✅ 핵심 기능(generative HUD 코어 루프)이 FE(React)–BE(Hermes `/v1` + orchestrator)에서 동작. ✅ 명령→스트리밍→HUD의 수직 슬라이스로 테스트 가능한 시나리오(e2e) 확보.
- 배운 것: **LLM 출력은 계약·자기치유·테스트로 감싸야 데모가 안 무너진다.** 빠른 생성보다 "깨져도 앱이 사는" 구조가 먼저다.
